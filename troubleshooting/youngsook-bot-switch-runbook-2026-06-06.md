# 영숙이 봇 전환 런북 — "비서 영숙"(@irir4r_secretary_bot)을 진짜 영숙이로 만들기 (2026-06-06)

> 목적: "영숙방에서 영숙이가 주식을 쓰게 한다"는 목표를 위해, 영숙방 무응답 원인을 추적하고 봇을 올바르게 교체하는 재사용 런북.
> 전제: 영숙이 코드 = `youngsook_listener.py` (비공개 레포 `comodo-ingong`, 루트 `/home/master/comodo-ingong`). 텔레그램 메시지를 받아 명령어별로 담당 로봇(주빵이/미미/코덱스 등)에 라우팅하는 "교환원".
> 🔒 토큰·키·chat_id 숫자는 이 공개 방에 적지 않는다 → 서버 `.env`에만.

## 0. 30초 요약 (TL;DR)

- 증상: 영숙방에서 `주빵`·`미미`·일반대화 **전부 무응답**. 그런데 **로봇방은 정상**.
- 진짜 원인: 봇이 죽은 게 아니라 **봇이 두 개**라서 생긴 혼동.
  - 코드(`youngsook_listener.py`)가 실제로 굴리는 봇 = **`@Irsar_bot`** (영숙방에서 주식 분석 정상 출력함).
  - 마스터가 "영숙"으로 기대한 봇 = **`@irir4r_secretary_bot`(비서 영숙)** — **이 코드가 안 굴리는 별개 봇** → 그래서 침묵.
- 결정(마스터): **비서 영숙을 진짜 영숙이로 만든다** (코드가 비서 영숙 봇을 굴리게 토큰 교체).
- ⚠️ 함정: 이 스크립트가 로봇방 말고 **뉴스방 등 다른 방도 담당**하면, 토큰만 바꾸면 그 방들도 같이 바뀜 → 그땐 **영숙방 전용 리스너 분리**가 정답.

## 0-2. 목표 아키텍처 — "한 방 = 한 봇" (2026-06-06 마스터 결정)

마스터: "뉴스방 봇은 따로 만들자." → 드라마처럼 **방마다 전용 봇 + 전용 리스너**로 분리하는 깔끔한 구조로 간다.

| 방 | 전용 봇 | 리스너 | 상태 |
|---|---|---|---|
| 영숙방(주식) | 비서 영숙 `@irir4r_secretary_bot` | secretary listener (신규) | **만들 것** |
| 드라마방 | DRAMA_BOT | `drama_listener.py` | ✅ 분리 완료 |
| 뉴스방(irsar_news) | 뉴스 전용 봇 (신규 발급) | news listener (신규) | **만들 것** |
| 귓뜸방 | 영숙 복제 | (영숙과 동일) | 추후 |

- 효과: 토큰 교체 위험 사라짐(각 리스너가 자기 봇·방만 봄). 한 방 죽어도 다른 방 안 죽음.
- 각 리스너 = `youngsook_listener.py`를 복사해 **자기 봇 토큰 + 자기 방 chat_id + 자기 방 트리거만** 보게 한 인스턴스.
- ⚠️ 우선순위: **주식(영숙방)이 최우선**(마스터 지시). 뉴스 분리는 그다음.
- 필요물: 신규 뉴스봇 토큰(BotFather), 비서 영숙 토큰, 각 방 chat_id (전부 🔒 서버 .env에만).

## 1. 진단 과정 (증상 → 원인, 시간순)

### 1차 — "주식 안 써짐"으로 출발
- 영숙이 코드 전문(1,621줄) 정독.
- 1순위 의심: **주빵이(juppang_analyzer) 모듈 경로 불일치** (import 39~44번 줄은 `_company` 없는 경로, watchlist 739번 줄은 `_company` 있는 경로). 둘 중 하나가 틀리면 `juppang_analyzer=None` → "모듈을 불러올 수 없습니다".
- 조치: 경로를 여러 후보 자동탐색하도록 고친 파일 제작·전달(py_compile 통과). *단 미배포.*

### 2차 — "주식만"이 아니라 "전부 무응답"
- 텔레그램 스크린샷: `주빵`·`미미`·일반대화 모두 무응답.
- 재해석: 주식 로직 문제 아님 → 봇 기동/수신/삼킴 문제.

### 3차 — "로봇방은 되는데 영숙방만 무응답"
- 봇이 죽은 게 아님(로봇방 정상). **방 단위 문제**로 확정.
- 코드 근거 — `youngsook_listener.py` 1314번 줄 (chat_id 보안 게이트):
  ```python
  if str(sender_id) != str(chat_id) and str(sender_id) not in {"-5012814805"}:
      → 외부접근 취급 → continue (무응답, 보안로그만)
  ```
  영숙이는 **설정값 `TELEGRAM_CHAT_ID` + 하드코딩 `-5012814805`** 딱 두 방에서만 응답.

### 4차 — 결정적: 영숙방에 봇이 두 개
- 스크린샷에서 영숙방에 **`@Irsar_bot`(주식 분석 답함) + `비서 영숙`(조용함)** 동시 존재 확인.
- multi-room-setup.md 13번 경고("코드 봇=@Irsar_bot, 비서 영숙=@irir4r_secretary_bot, 헷갈리지 말 것")가 실제로 발생.
- **결론: 영숙방은 사실 '되고' 있었고(Irsar_bot이 답함), 마스터가 본 '비서 영숙' 봇이 코드와 연결 안 된 봇이었던 것.**

## 2. 전환 절차 — 토큰 교체 방식 (영숙이 스크립트가 영숙방/로봇방만 담당할 때)

> ⚠️ 시작 전 확인: 이 스크립트가 **뉴스방 등 다른 방도 답하면 이 방식 쓰지 말 것** (3장으로). 뉴스 트리거(`뉴스 [주제]`)가 이 코드에 있어 다방 담당 가능성 있음.

**STEP 1 — BotFather에서 '비서 영숙' Group Privacy 끄기** ⭐가장 잘 빠뜨림
- @BotFather → `/mybots` → `@irir4r_secretary_bot` → **Bot Settings → Group Privacy → Turn off**
- 안 끄면 "주빵 삼성전자" 같은 **일반 메시지를 못 읽음**(슬래시 명령만). → 무응답 단골 원인.

**STEP 2 — 영숙방에 비서 영숙 봇 넣고 관리자 지정**
- 영숙방 → 멤버 추가 → `@irir4r_secretary_bot` 초대 → **관리자(admin)** 지정
- (권장) **Irsar_bot은 영숙방에서 내보내기** — 둘 다 답하는 혼란 방지

**STEP 3 — 영숙방 chat_id 잡기**
- 비서 영숙 봇 넣은 직후 영숙방에 아무 말 전송 → 서버에서 `getUpdates`로 `chat.id` 확인(런북 multi-room STEP2), 또는 재기동 후 콘솔 `[DEBUG chat_id] 수신 chat_id=...` 숫자. (그룹은 `-100…`)

**STEP 4 — 서버 `.env` 수정** (🔒 토큰은 공개 채팅·메모리룸에 붙이지 말 것)
- `TELEGRAM_BOT_TOKEN` = 비서 영숙 봇 토큰
- `TELEGRAM_CHAT_ID` = 영숙방 chat_id

**STEP 5 — 재기동 + 테스트**
- 영숙이 재기동 → 영숙방에 `주빵 삼성전자` → **비서 영숙**이 답하면 성공 ✅
- 기동 로그에 `juppang_analyzer 로드 성공/실패`도 같이 확인(주식 모듈 경로).

## 3. 전환 절차 — 전용 리스너 분리 방식 (영숙이 스크립트가 뉴스방 등 여러 방을 담당할 때)

> 드라마봇 분리(drama_listener.py)와 동일 패턴. 기존 youngsook_listener는 건드리지 않고 영숙방만 떼어냄.

1. `youngsook_listener.py` 복사 → `secretary_youngsook_listener.py`(가칭) 생성.
2. 새 리스너가 **비서 영숙 토큰 + 영숙방 chat_id**만 보도록 `.env` 변수 분리 (예: `SECRETARY_BOT_TOKEN`, `SECRETARY_ROOM_CHAT_ID`).
3. 1314번 chat_id 게이트의 하드코딩 `-5012814805` 제거 또는 영숙방 값으로 교체.
4. 기존 youngsook_listener에서 영숙방 chat_id 제거(중복 응답 방지).
5. 비서 영숙 봇 Group Privacy off + 영숙방 관리자 지정(2장 STEP1·2 동일).
6. 새 리스너 프로세스 별도 기동 + 검증(`pgrep -f`로 프로세스 수 확인).

## 4. 미해결/확인 대기 항목

- [x] **이 스크립트가 뉴스방 등 다른 방도 답하나?** → ✅ **확인됨: 그렇다.** 채팅 목록 스크린샷(2026-06-06)에서 `irsar_news` 방의 최신 메시지가 `Irsar_bot:`로 시작 → **Irsar_bot이 뉴스방도 담당.** → **토큰 교체(2장) 금지, 전용 리스너 분리(3장)가 정답으로 확정.**
- [x] **"비서 영숙"은 방인가?** → ❌ **아니다. 봇과의 1:1 대화창**(그룹 아님). "영숙방"이라 부를 그룹이 아님. 주식 분석은 실제로 **"Irsar_bot" 대화**에서 나오는 중(=주식은 이미 작동).
- [ ] 주빵이(juppang_analyzer) 모듈 경로 정상 로드 여부 (기동 로그).
- [ ] 하드코딩 chat_id `-5012814805`의 정체(어느 방인지) — 로봇방/뉴스방 중 하나로 추정.
- [ ] `room_pipeline.py`(매 메시지 먼저 가로채는 모듈) 미수령 — 별도 라우팅 영향 확인 필요.
- [ ] "자동 감시"(시세 주기 알림)는 이 파일에 없음 → 별도 스케줄러 프로그램 확인.

| 채팅 | 정체 | 담당 봇 | 비고 |
|---|---|---|---|
| irsar_drama | 드라마방(그룹) | DRAMA_BOT | drama_listener.py |

### ✅ 목표 확정 (2026-06-06, 마스터)
- **마스터: "영숙방은 @irir4r_secretary_bot 이 로봇이네."** → **영숙방 = 비서 영숙 봇이 있는 곳**으로 확정.
- **최종 목표: 비서 영숙(@irir4r_secretary_bot)이 영숙방에서 주식을 답하게 한다.**
- **방법 확정: 전용 리스너 분리(3장).** Irsar_bot/뉴스방은 절대 안 건드림(토큰 교체 금지).
- 비서 영숙 봇 상태: Group Privacy **이미 OFF**(확인됨), 공통 그룹 1개.
- 실행 형태: 기존 `youngsook_listener.py`를 **두 번째 인스턴스**로 돌리되, 그 인스턴스의 설정(`telegram_private.json` 또는 전용 .env 변수)이 **비서 영숙 토큰 + 영숙방 chat_id**를 보게 한다. (juppang/persona 상대경로 깨짐 주의 → 경로 자동탐색 패치 필요.)
- 미결정: 영숙방이 **그룹인지 1:1 대화인지** (리스너 동작엔 영향 적음, 다만 여러 사람 사용 시 그룹 권장).

→ 경로 자동탐색 수정 파일도 **봇이 안 켜져 있으면 무의미.** 먼저 "살아있는가"부터 확인해야 함.

### 전부 무응답일 때 코드상 의심 지점 (youngsook_listener.py)
1. **봇 프로세스 자체가 안 돎** (크래시/종료/서버 리부트). main 루프는 per-message 예외는 5초 sleep 후 계속하지만, 기동 자체 실패면 침묵.
2. **chat_id 인증 게이트 (1314번 줄):** 수신 방의 chat_id가 설정값 `TELEGRAM_CHAT_ID` 도 아니고 하드코딩 `-5012814805` 도 아니면 → 조용히 `continue`(답 없음). 방이 바뀌었거나 설정이 어긋나면 그 방 전체가 침묵.
   - 단, 코드가 수신 시 `[DEBUG chat_id] 수신 chat_id=... text=...`를 매번 print → **콘솔만 보면 수신 여부 즉시 판별 가능.**
3. **room_pipeline 가로채기 (1326~1333번 줄):** 매 메시지에서 `room_pipeline.room_dispatch()`를 **먼저** 호출. 이게 True(처리함)를 리턴하면 `process_message` 건너뜀. room_pipeline이 주빵/미미를 "처리했다"고 하고 **실제 답은 안 보내면 → 조용히 삼켜짐.** ← 선택적 무응답의 유력 후보. **단 `room_pipeline.py`는 미수령(별도 파일).**
4. **자동 감시는 이 파일에 없음:** watchlist 등록/조회/해제만 있고, 주기적으로 시세 보고 알림 쏘는 루프는 별도 프로그램(stock_expert 스케줄러)일 것 → 그 프로그램도 따로 떠 있어야 함.

### 무응답 원인 가르는 단 하나의 결정적 행동
**서버에서 영숙이 재기동 → 검은 콘솔 띄워둔 채 영숙방에 `주빵` 전송 → 콘솔에 뜨는 글자 확인:**
- `[DEBUG chat_id] 수신 chat_id=...` **뜸** → 봇은 받고 있음 → 침묵은 하류(인증게이트 or room_pipeline). 그 chat_id 값과 설정값 비교 / room_pipeline 점검.
- **아무것도 안 뜸** → 봇이 안 받음(미기동 / 봇토큰 다름 / 네트워크). → 기동 로그 확인.
- 기동 자체 에러 → 그 에러가 곧 원인.

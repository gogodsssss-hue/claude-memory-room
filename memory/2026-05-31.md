## [사건] 주빵이 "분석해" 무응답

증상
- 텔레그램에 "주빵아 OO 분석해" → 무응답. "분석 중입니다…" 안내조차 안 옴.
- 반면 코덱스, 일반 메시지는 정상 응답.

진단 과정에서 드러난 곁다리 문제들
1. run_bot_forever.sh 감시자가 5개까지 누적 실행 → 리스너 다중 → 같은 토큰 동시 폴링으로 텔레그램 409 충돌 + Gemini 키 429(5배 호출로 한도 조기 소진).
2. cron 이중 감시(가드 + bot_watchdog.sh) 충돌.
3. .env 22·24줄 한글 메모가 키 형식이 아니라 python-dotenv 파싱 경고. (키 로드엔 무해)
4. 메인 폴링 루프가 except Exception: time.sleep(5)로 모든 예외를 출력 없이 삼킴 → 원인 불명 무응답처럼 보임.

진짜 원인 (최종)
- telegram_listener.py에 중복으로 들어간 2번째 코덱스 처리 블록(773~781줄)에서 변수명을 message로 오타. 실제 변수명은 message_text.
- 코덱스는 위쪽에서 먼저 처리되어 일찍 return → 이 줄 안 옴(정상).
- 주빵이 등 다른 메시지는 이 줄 통과 시 NameError: name 'message' is not defined 발생 → 메인 루프 except가 조용히 삼킴 → 무응답.

해결
- 해당 4곳 message → message_text 로 교체. (단 res["message"] 같은 딕셔너리 키는 건드리지 않음)
- 감시자/리스너 전멸 후 단일 인스턴스로 재기동.
- cron은 가드 1줄 + @reboot만 유지(이중 감시 bot_watchdog 제외).
- 메인 루프 except에 traceback.print_exc() 추가 → 앞으로 예외가 로그에 보이게.
- 수정 후 "주빵아 삼성전자 분석해" → "분석 중입니다…" 정상 출력 → 분석 리포트 도착 확인.

교훈
- "무응답"은 대개 예외 삼키기 + 변수 오타 또는 리스너 다중 실행이 원인.
- 디버깅 1순위: 메인 루프의 except가 에러를 출력하게 만든 뒤 foreground로 재현.
- 멈춤(hang)으로 보이면 py-spy dump --pid로 스택을 떠본다. 단, 덤프가 getUpdates(폴링)를 가리키면 처리를 끝내고 다음 메시지를 기다리는 정상 idle → 핸들러가 응답 없이 끝난 것이니 예외 가시화로 돌아간다.
- 같은 기능을 두 번(중복 블록) 넣으면 변수명 불일치 같은 사고가 숨는다. 중복 핸들러는 제거 권장.
